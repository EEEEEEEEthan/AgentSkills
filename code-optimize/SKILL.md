# 代码优化

## 核心目标

**终极目标：减少维护成本**

**子目标：减少类成员**

## 重要原则

将一个很长的方法拆开并不符合目标。除非这个方法的某些代码片段能够被复用。

## 优化检查清单

执行代码优化时，按以下清单逐项检查：

- [ ] 移除冗余逻辑（未被调用过的逻辑）
- [ ] 封装，复用重复代码片段
- [ ] 仅被引用过一次的方法，应该成为内联实现或者成为内部方法
- [ ] 仅被一个方法调用过多次的方法，应该成为内部方法
- [ ] 高频方法里的内部方法应该尽量用 static
- [ ] 命名规范：
  - C#: 变量/常数/内部方法用 lowerCamelCase，属性/方法/事件用 CamelCase，事件需 On 前缀，不要有下划线前缀
  - GDScript: 类名用 UpperCamelCase，常量用 ALL_UPPER_CASE，其他（变量/函数/方法）用 snake_case
- [ ] 属性可以改为 field 关键字的要用 field。例如：
  - `[field: MaybeNull] NodeType NodeName => field ??= GetNode<NodeType>("%NodeName")`
  - `NodeType? NodeName => field ??= GetNodeOrNull<NodeType>("NodeName")`
  - 属性非空，field 可能为空需要增加 `[field: MaybeNull]` 标签
  - field 语法在一些 IDE 里会被误报语法错误，但他是正确的语法
- [ ] 移除所有多余空行
- [ ] 移除所有行内注释（保留文档注释）
- [ ] 旧的文档注释需要检查正确性。不需要增加新的文档注释
- [ ] 跨帧的状态应该封装为 IEnumerator，嵌套 while 表达子状态，用 `_process` 推进
- [ ] 完全信任 nullable 语法警告，不做任何多余的防御代码
- [ ] 用最新的语法编写扩展方法/属性。这也在一些 IDE 里会被误报语法错误，但他也是正确的

## 优化策略

### 减少类成员

优先考虑：
1. 内联单次调用的方法
2. 将仅被一个方法多次调用的方法改为内部方法
3. 移除未使用的成员和逻辑

### 代码复用

识别重复代码片段，提取为可复用的方法或属性。但避免为了拆分而拆分——只有真正可复用的代码才值得提取。

### 性能优化

- 高频方法中的内部方法使用 `static` 减少实例开销
- 使用 `field` 关键字优化属性访问
- 跨帧状态使用协程（IEnumerator）而非每帧检查

### 代码清理

- 移除多余空行
- 移除行内注释（保留文档注释）
- 检查并更新过时的文档注释
