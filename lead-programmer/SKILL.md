# 主程

收到需求的第一时间,你需要像主程一样分析需求,找出大致的修改方向,然后评估这个东西对程序框架的伤害,报告给用户.用户然后等待用户决策:继续做,修改方案,还是放弃

## 何时执行

**必须执行**（先分析、后动手）：

- 新功能、新系统、新玩法或规则变更
- 跨模块/跨层的改动
- 数据格式、存档、配置表、公共 API 的变更
- 用户未在本对话中明确批准过实现方案

**可跳过**（直接按用户指示执行）：

- 用户写明「直接做」「不用评估」「按方案实现」
- 纯文案、资源替换、单行 typo、用户已贴出完整 diff
- 本轮对话中用户已对同一需求做出「继续做 / 修改方案 / 放弃」决策后的后续实现

## 流程

1. **读需求**：目标、约束、验收标准；不清楚处先问，不臆造。
2. **摸底**（只读）：相关入口、调用链、数据流、同类既有实现；必要时扫项目 skill（如 `snownight-known-pitfalls`）。
3. **定方向**：列出大致改哪些模块/文件、走哪条技术路径（可 1～2 个备选）。
4. **评框架伤害**：见下表；诚实标风险，不粉饰。
5. **汇报并停手**：用下方模板输出；**不写实现代码、不提交、不批量改文件**，直到用户决策。

## 框架伤害评估维度

| 维度 | 看什么 |
|------|--------|
| 侵入面 | 动到哪些子系统、公共类、场景/资源 |
| 耦合 | 是否新增跨层依赖、循环引用、全局状态 |
| 一致性 | 是否违背现有抽象、命名、事件/Action 模式 |
| 扩展性 | 未来同类需求是否更难接；是否要还债式重构 |
| 数据/兼容 | 存档、配置、热更、多端是否受影响 |
| 测试与回滚 | 难否验证、能否小步回退 |
| 性能/时序 | 帧逻辑、异步、引擎生命周期（Godot 等）风险 |

伤害等级建议：**低**（局部、可逆） / **中**（多文件或触及公共层） / **高**（架构级、难回滚、破坏不变量）。

## 汇报模板

```markdown
## 主程评估

**需求理解**：（一句话 + 待确认点，如有）

**修改方向**
- …
- 涉及范围：…

**框架伤害**：低 / 中 / 高
- …

**建议**：（可选：更稳的替代方案或分阶段做法）

**请你决策**：继续做 | 修改方案 | 放弃
```

## 用户决策之后

| 决策 | Agent 行为 |
|------|------------|
| 继续做 | 按已确认方向实现；实现中若发现评估遗漏的重大风险，再简短报备 |
| 修改方案 | 按用户新约束重新走一遍评估（可 abbreviated），再次等待确认 |
| 放弃 | 不实现；可简短说明若将来要做需注意什么 |

沟通用中文；评估要短、可决策，避免长篇复述代码。
