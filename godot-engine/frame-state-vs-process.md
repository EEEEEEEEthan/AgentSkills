# 避免依赖帧循环修正状态

## 问题

把「本该立刻一致」的逻辑放在 `_process` / `_physics_process`、或默认拖到下一帧再改 UI / 碰撞体 / 可见性时，常见后果不是单纯 CPU 浪费，而是：

- **错一帧再改对**：用户会看到一帧的错位、错误贴图、错误碰撞或错误数值，表现为闪烁或抖动。
- **时序难测**：帧顺序、delta、暂停与 `time_scale` 会让问题偶发，排查成本高。

## 默认做法

- **在数据或权威状态变更的同一处**完成依赖更新（发信号、调用 setter、在伤害结算后立刻改条等），让表现与数据同一步调。
- 若与布局/尺寸相关，与 [layout-refresh.md](layout-refresh.md) 一致：需要本帧结果就**显式**触发布局或排序，而不是「每帧在 process 里猜什么时候对了再写」。

## 何时才考虑放进帧循环

- 持续跟随型逻辑（如每帧跟目标的相机插值）、输入采样等**本来就需要每帧**的行为。
- 不是「上一帧算错了这一帧补一下」的补丁式矫正。

## 判断标准

若某段 process 逻辑只是在「等状态稳定后再对齐一次」，优先改为在状态变化源上**同步**处理；仍不够再考虑单次 `call_deferred` 等明确延后，并写清原因，而不是长期挂在帧更新里。
