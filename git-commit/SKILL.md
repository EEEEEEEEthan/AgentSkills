# Git Commit

## 目标

在 Windows + PowerShell 环境中，安全完成一次提交，确保提交内容覆盖：
- 本次工作修改
- 当前文件变更
- 用户已 `git add` 的内容

## 执行规则

1. 使用 PowerShell 执行 Git 指令。
2. 先检查状态与差异，再决定补充暂存哪些文件。
3. 保留用户已暂存内容，不清空、不覆盖。
4. 仅把本次工作相关文件补充 `git add`。
5. 提交信息必须基于暂存区变更编写，并结合本次工作目标说明“为什么改”。
6. `git commit` 必须追加 `--trailer` 签名。
7. 如遇符号链接路径，先确认目标是否为预期仓库内容，再决定是否纳入提交。

## 标准流程

按顺序执行：

1. 查看状态与变更
   - `git status --short`
2. 判断本次工作涉及文件，补充暂存（不动无关文件）
   - `git add <相关文件>`
3. 生成提交信息前，再次核对暂存区并据此撰写
   - 由模型自行选择查看暂存区的方式，避免二进制文件或超大 diff 带来的上下文膨胀
   - 提交信息以暂存区变更为唯一提交事实来源，未暂存内容仅作上下文参考
4. 执行提交（带 trailer）
   - `git commit -m "<中文提交信息>" --trailer "<你的签名键>=<你的签名值>"`
5. 提交后复查
   - `git status`

## 提交信息模板

格式要求：

1. 第一行：一句话主题，30 个汉字以内，不需要类型前缀。
2. 第二行：空行。
3. 第三行起：补充细节，说明关键改动与目的。

示例：
`修正敌人意图动画提交遗漏

补齐相关文件的暂存范围，确保提交内容与实际功能修改一致。`

## 质量门槛

- 提交前确认无明显无关文件被暂存。
- 提交信息与暂存区事实一致，不引用未暂存改动作为已提交事实。
- 提交后确认工作区状态符合预期。
- 若存在不确定的符号链接文件，先停止并向用户确认。
