# 约定：简洁优先于「最小 diff」，并对旧稿保持核对

## 规则：少绕弯、少封装噪音

- **改动行数少**不是目标本身；在等价行为下，优先选**更直、更少间接层**的写法。
- 若不存在额外不变式、校验或对外 API 约束，**不要**用「私有字段 `_x` + 只读属性 `x`」仅为了避免在构造函数参数里与成员同名；可统一成公有字段 `x`，构造函数参数用 `p_x` 等区分并赋值。
- 避免为了「少碰几行」而保留绕弯结构，增加后续阅读与修改成本。

## 规则：旧稿是线索不是结论

- **既有代码、项目习惯、引擎占位用法**须核对语义、生命周期与产品意图，避免「一直这样写就接着堆」。
- **改完后的结果**要在当前上下文里经得起「这是不是最直、责任最清晰的一版」：控制流扁平、副作用与技巧集中在窄边界。不为「diff 看起来小」在外层加循环或换入口来**回避**局部根因，除非比较过确无更干净的等价方案。
- 「清晰」指**当下问题**的解；与 [extensibility.md](extensibility.md) 不矛盾——不为未发生需求预建框架。

## 不符合（仅为少改字段名）

```gdscript
var _side: Defs.Side
var side: Defs.Side:
	get:
		return _side
func _init(..., side: Defs.Side) -> void:
	_side = side
```

## 符合（直接字段 + 构造参数区分）

```gdscript
var side: Defs.Side
func _init(..., p_side: Defs.Side) -> void:
	side = p_side
```

## 不符合（旧稿与收工）

明知旧结构在回避某个边界问题，仍只在更外层加循环或复制入口，使调用链变长、读者更难一次看懂数据从哪来到哪去。

## 符合（旧稿与收工）

先确认问题根因；能在一处用更小改动消掉技巧或重复，就改那一处；若必须动编排，则新结构一眼可读且旧坑不再扩散到全局。

## 说明

当确实需要 setter 副作用、序列化边界或子类约束时，再引入封装；否则保持数据与命名一目了然。
