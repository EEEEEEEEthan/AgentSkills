# 约定：预留合理扩展性

## 规则

- **先满足当前需求**，在「很可能变」的边界留缝：枚举/策略对象/小接口、数据驱动配置，而不是抽象层叠满全局。
- **扩展点要窄**：一处编排（如战斗结算、关卡加载）比到处 `if type ==` 更易演进；新类型优先接同一套协议而非复制粘贴一整条流程。
- **避免过度设计**：不要为了「万一」引入大量空子类与中间层；当只有两种实现时，分支或简单策略往往足够，等第三种出现再抽公共形状。

## 不符合

为每个字段写 getter/setter 钩子「以备将来」、或引入未使用的插件式接口，导致阅读与跳转成本上升却无实际分支。

## 符合

在稳定边界上命名清晰、把变化收敛到单文件或单模块；需要插件化时再抽出接口与注册表，并与现有调用方对齐测试。
