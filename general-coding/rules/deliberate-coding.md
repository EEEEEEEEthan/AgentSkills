# 约定：写代码要慎重

## 规则

- **先想清楚再改**：入口顺序、并发/重入（信号链里再改同一状态）、成功与失败分支是否都一致；避免「先糊一版再靠注释或后续补丁圆」。
- **改动面要可控**：一次提交解决一类问题；跨模块大改时先确认调用约定与生命周期，避免留下半套迁移状态。
- **与「最少注释」一致**：不靠长注释解释为何危险，而用更小、可证明正确的步骤与命名表达约束；非常识性坑位用**最短**注释标出即可（见 [minimal-comments.md](minimal-comments.md)）。

## 不符合

在未理清调用顺序的情况下，在多处各写一段「差不多能跑」的特殊判断，彼此还可能打架。

## 符合

先列出状态变迁或调用链，在边界或单一编排点集中处理；必要时加最小测试或自测路径验证关键分支。
