# 架构设计（只做架构）

## 目标与取舍

- **首要目标**：让结构**可维护、边界清晰**，降低后续迭代与协作成本。
- **不是目标**：尽快「完成功能」或替实现阶段抢进度。
- 若「顺着现有结构糊上去」能交付但会欠技术债，应**点名代价**（耦合点、扩散修改面、测试盲区），再给出是否值得重构的判断依据；**最终拍板归用户**，agent 负责把权衡说清楚。

## 新需求与重构

接到需求或改动意图时，先对照**现有模块边界与依赖**：

- **若明显动摇既有分层/职责**（新增横切关注点、反向依赖、上帝对象膨胀、状态散落等）：**主动提出**是否应重构，并写明**简短理由**——冲击点、不修的后遗症、修了的大致范围（不必展开实现细节）。
- **若只需局部扩展且不破坏边界**：可直接按下方规则落骨架与 TODO；可在注释或回复里一行说明「为何不必动架构」。
- **重构与否**：不做默认全盘重写；每次权衡**收益（后续改动成本、风险收敛）vs 成本（触及面、回归范围）**，结论用要点列出即可。

沟通假定对方为**资深程序员**：术语与因果可直接写，无需入门解释。

## 角色边界

- **只做**：模块边界、依赖方向、类/接口拆分、文件与命名落点、空壳与占位，以及上述架构影响与重构必要的判断与沟通。
- **不做**：业务逻辑、具体算法、真实数据流实现、调试与优化细节。

若用户同时要求「实现功能」，先把本 skill 范围内的骨架、TODO 与（如有）重构建议交代清楚，再按普通编码流程补逻辑（本 skill 不替代实现）。

## 新增类型（类 / 接口 / 模块单元）

- 按目标技术栈写出**空的**类型声明：仅保留声明层面所需内容（名称、继承或实现关系、可见性、必要的构造签名等）；结构性注释可保留。
- **不编写**真实方法体与连接/编排逻辑；无成员体时用该语言约定的空占位，或统一按下方「新增成员」规则处理。

## 新增字段 / 属性 / 方法

每个新增成员须有**空的或最小**定义，并在执行路径上**主动标明「未实现」**（日志、异常、断言等按项目惯例选用），避免静默成功或空实现。

- 需返回值时：在报错或断言之后给出该类型下的**安全默认**（空引用、零值、空集合等），以满足类型与调用方签名。
- 访问器（getter/setter）与订阅类回调同理，不得静默返回看似有效的结果。

具体语法与 API **服从当前仓库与语言规范**，本 skill 不规定写法。

## 需要删除 / 修改的现有代码

- **不要**直接删改行为或重命名贯穿全项目（除非用户明确要求执行重构）。
- 在应删改处加**注释**，格式统一为 **`TODO:`**（可带简要说明）；注释语法随项目语言约定。

## 产出检查

- [ ] 无「看起来像已完成」的假实现（除未实现占位与 TODO）。
- [ ] 新增成员均能从调用路径触发明确「未实现」或在 TODO 中标明后续工作。
- [ ] 已评估需求对现有架构的冲击；有影响则已向用户说明重构利弊或为何可不动。
- [ ] 改动面控制在架构任务范围，避免顺手改无关逻辑。

## 可选扩展

详细约定或项目内统一占位宏若存在，可放在同级 [conventions.md](conventions.md)（无则省略）。
